# 制造业设备报修文本故障分类与停机风险预警项目方向整理

## 1. 项目定位

本项目面向制造业设备运维、工厂维修、工业互联网和售后服务场景，核心目标是对一线员工或设备操作员提交的报修文本进行自动理解，完成故障类型分类、停机风险判断和维修处理部门推荐。

项目名称可以定为：

> 面向制造业设备运维的报修文本故障分类与停机风险预警系统

也可以在简历中写成：

> 基于 BERT 与知识蒸馏的制造业设备报修文本分类与停机风险预警系统

该项目不是泛泛的文本分类练习，而是围绕制造企业设备维修中的真实业务痛点设计。企业在生产过程中会积累大量设备报修、维修工单、报警日志和点检记录，其中大量信息以自然语言文本形式存在。如果这些文本只能依赖人工查看、人工分派、人工判断风险，就会造成响应慢、派单错误、停机预警不及时等问题。

## 2. 真实业务问题

制造企业常见问题包括：

- 一线员工提交报修文本后，维修调度人员需要人工判断故障类型。
- 报修描述短、口语化、不规范，例如“机床有异响”“传感器没反应”“PLC 报警 E102”。
- 同一类故障可能有多种表达方式，人工分派容易不稳定。
- 高风险停机事件没有被及时识别，导致产线停机时间扩大。
- 历史维修经验沉淀在工单文本中，但没有被有效复用。
- 不同部门之间，如机械、电气、自动化、仪表、安全部门，容易出现派单错位。

因此，该项目可以帮助企业提升设备维修响应效率，降低误派率，并对可能导致停机的故障提前预警。

## 3. 典型输入与系统输出

示例输入：

| 报修文本 | 故障大类 | 故障部位 | 风险等级 | 推荐部门 |
|---|---|---|---|---|
| 1 号 CNC 主轴启动后异响，转速上不去 | 机械故障 | 主轴系统 | P1 | 机械维修 |
| 包装线传感器无信号，物料检测不到 | 传感器故障 | 光电检测 | P1 | 自动化/仪表 |
| PLC 报警 E102，设备无法复位 | 控制系统故障 | PLC | P0 | 自动化/电气 |
| 空压机温度过高自动停机 | 工艺/热异常 | 温控系统 | P0 | 设备维修 |
| 皮带跑偏，偶尔卡料 | 机械故障 | 输送系统 | P2 | 机械维修 |
| 急停按钮松开后设备仍不能启动 | 安全风险 | 安全回路 | P0 | 安全/电气 |

项目最终输出不应只是一个类别，而应该是结构化结果：

```text
输入文本：空压机温度过高自动停机
故障大类：工艺/热异常
故障部位：温控系统
停机风险：P0
推荐处理部门：设备维修/电气
模型置信度：0.93
是否建议人工复核：否
```

## 4. 推荐标签体系

### 4.1 一级故障分类

| 一级分类 | 说明 |
|---|---|
| 机械故障 | 异响、卡滞、磨损、松动、跑偏、断裂、漏油 |
| 电气故障 | 电机、接触器、变频器、电源、短路、过载 |
| 传感器故障 | 光电、接近开关、温度、压力、液位、编码器 |
| 控制系统故障 | PLC、HMI、伺服、通讯、程序、报警码 |
| 工艺/参数异常 | 温度、压力、速度、张力、流量等工艺参数偏离 |
| 耗材/备件问题 | 刀具磨损、滤芯堵塞、皮带老化、润滑不足 |
| 安全风险 | 急停、安全门、联锁、漏电、过温、烟雾 |
| 预防性维护 | 点检、保养、校准、润滑、周期换件 |

### 4.2 故障部位分类

| 设备类型 | 常见故障部位 |
|---|---|
| CNC 机床 | 主轴、刀库、伺服轴、冷却系统、润滑系统 |
| 包装机 | 输送带、封口机构、传感器、热封模块、气缸 |
| 空压机 | 电机、油路、温控、压力系统、滤芯 |
| 注塑机 | 液压系统、加热圈、模具、锁模机构 |
| 工业机器人 | 伺服、关节、示教器、通讯、夹具 |
| 输送设备 | 皮带、滚筒、电机、光电、气动元件 |

### 4.3 停机风险等级

| 等级 | 含义 | 示例 |
|---|---|---|
| P0 | 已停机或极高概率停机 | 设备无法启动、主线停了、PLC 无法复位 |
| P1 | 高风险，短时间可能停机 | 温度持续升高、异响明显、传感器频繁误报 |
| P2 | 中风险，影响效率或质量 | 偶尔卡料、速度不稳定、产品偏位 |
| P3 | 低风险，咨询或计划维护 | 需要更换滤芯、申请定期点检 |

## 5. 项目核心任务

建议将项目设计为多任务文本分类系统，而不是单一分类任务。

1. 故障类型分类：判断文本属于机械、电气、传感器、控制系统等哪类故障。
2. 故障部位识别：识别主轴、电机、PLC、传感器、皮带、气缸等关键部位。
3. 停机风险分类：判断故障风险等级 P0/P1/P2/P3。
4. 维修部门推荐：根据故障类型和风险等级推荐维修处理部门。
5. 相似历史工单召回：根据当前报修文本召回相似历史维修记录，辅助维修人员快速定位问题。

## 6. 数据来源思路

制造业真实报修工单一般存储在企业内部系统中，公开互联网上很难直接获得十几万条真实中文工单。因此，项目的数据来源需要设计得可信、可解释。

### 6.1 企业内部真实数据

如果是企业内部项目，数据通常来自：

| 数据源 | 主要内容 |
|---|---|
| CMMS/EAM 设备管理系统 | 报修工单、维修描述、维修结果、停机时长 |
| MES 制造执行系统 | 产线、工序、设备状态、停线记录 |
| SCADA/DCS/PLC 报警日志 | 报警码、报警时间、点位信息、复位状态 |
| 点检系统 | 巡检记录、异常描述、保养记录 |
| 备件系统 | 更换零件、耗材消耗、库存信息 |
| QMS 质量系统 | 设备异常导致的不良品、返工、报废记录 |
| 售后服务系统 | 设备厂商维修记录、客户报修记录 |

### 6.2 个人项目可用数据来源

如果是个人简历项目，建议使用“公开工业数据 + 规则化构造 + 受控文本改写 + 人工抽检”的方式构建语料。

可用来源包括：

| 来源 | 用途 |
|---|---|
| UCI AI4I 2020 Predictive Maintenance Dataset | 提供结构化预测性维护数据和故障标签 |
| NASA Prognostics Center 数据集 | 提供设备退化、运行至失效、剩余寿命预测相关数据 |
| PHM Society Data Repository | 提供工业故障诊断和预测性维护竞赛数据 |
| MIMII 工业异常声音数据集 | 提供风扇、泵、阀门、滑轨等异常机器场景 |
| Kaggle 工业维护合成数据 | 提供大规模维护工单和传感器标签数据 |
| 设备说明书和报警码手册 | 提供 PLC、变频器、CNC、空压机等故障码、故障原因 |
| 工业维修案例文章 | 提供真实表达风格和维修过程描述 |

## 7. 十几万条数据如何构建

推荐构建方式：

| 数据来源 | 目标数量 | 说明 |
|---|---:|---|
| 公开维护工单或合成维护工单数据 | 5 万到 10 万 | 作为主训练语料 |
| 结构化预测性维护数据转文本 | 2 万到 5 万 | 使用真实字段生成报修描述 |
| 设备报警码和维修手册转文本 | 1 万到 3 万 | 增强专业故障码场景 |
| 工业维修案例人工整理 | 3000 到 10000 | 用作高质量验证集和测试集 |
| LLM 受控改写增强 | 3 万到 8 万 | 只改写表达，不改变故障事实和标签 |

关键点是不能简单说“用大模型生成数据”。更可信的说法是：

> 基于设备类型、故障部位、报警码、传感器状态、停机状态、维修结论等结构化事实生成多种自然语言报修表达，并通过人工抽检保证标签一致性。

## 8. 数据不足时的解决措施

1. 弱监督标注：通过关键词、报警码、设备部位规则自动生成初始标签。
2. 模板化生成：建立设备、部位、故障现象、风险等级词典，组合生成报修文本。
3. LLM 受控改写：只让模型改写表达方式，不允许改变故障事实和标签。
4. 主动学习：让模型挑选低置信度样本，人工优先标注这些难例。
5. 半监督学习：用 teacher 模型给未标注样本打伪标签，只保留高置信度样本。
6. 分层建模：先做故障大类，再做细分类，降低每个类别的数据量压力。

## 9. 面试时的数据来源回答

可以这样回答：

> 这个项目没有伪装成某工厂私有数据。我构建的是半合成工业运维文本分类数据集。数据底座来自公开预测性维护数据集、PHM 工业故障数据、设备报警码手册和公开维修案例。我先把结构化故障记录转换成报修文本，再用设备类型、故障部位、报警码、停机状态生成多层级标签。为了避免数据失真，我保留人工审核样本作为测试集，并做了去重、类别均衡和标签一致性检查。

简历表述可以写成：

> 构建 15 万条设备报修文本语料，数据来源包括公开预测性维护数据、设备故障手册、工业维修案例与规则化增强样本；设计机械、电气、传感器、控制系统、安全风险等多层级标签体系，并完成数据清洗、脱敏、弱监督标注和人工抽检。
