# 制造业设备报修文本故障分类与停机风险预警项目实施方案

## 1. 项目定位

项目名称：

> 基于 BERT 与知识蒸馏的制造业设备报修文本故障分类与停机风险预警系统

项目核心目标：

> 将一线员工、设备操作员、班组长、售后人员提交的设备报修文本，自动识别为故障大类、停机风险等级和推荐处理部门，从而提升维修派单效率、降低停机损失，并沉淀企业设备运维知识。

如果类比“新闻文本分类”的主要价值是提升点击率、订阅量和付费量，那么本项目的主要价值是：

- 提升维修工单分派准确率，减少工单反复转派。
- 缩短 MTTR，即平均维修恢复时间。
- 降低非计划停机时间，减少产线停产损失。
- 提升 OEE，即设备综合效率。
- 提前识别 P0/P1 高风险工单，降低安全事故和批量质量异常风险。
- 沉淀历史维修知识，为后续智能运维、预测性维护和备件管理提供数据基础。

本项目不是泛泛的文本分类练习，而是制造业数字化、工业互联网、智能运维、设备售后和企业服务中真实存在的工程问题。

## 2. 项目背景

制造企业的生产现场每天会产生大量设备相关记录，包括报修工单、点检记录、报警日志、维修记录、备件更换记录、停机记录和售后服务记录。很多关键信息不是标准字段，而是自然语言文本，例如：

```text
1号包装线光电传感器无信号，物料检测不到，设备频繁停机。
CNC主轴启动后异响，转速上不去，怀疑轴承磨损。
PLC报警E102，自动模式无法运行，现场无法复位。
空压机温度持续偏高，保护停机，影响整条产线供气。
冲压机安全门关闭后仍报警，急停复位后设备无法启动。
```

这些文本看似短，但包含设备、部件、故障现象、疑似原因、风险程度和处理部门等信息。传统方式通常依赖人工调度员阅读文本后，再判断应该派给机械、电气、自动化、仪表、安全或设备保养人员。

人工处理存在的问题：

- 报修文本口语化、不规范，人工判断标准不稳定。
- 同一故障有多种表达，例如“没反应”“无信号”“检测不到”“一直不触发”。
- 严重停机风险容易被普通工单淹没。
- 工单转派错误会拉长维修响应时间。
- 历史维修经验没有结构化沉淀，后续难以复用。
- 管理层难以及时统计高频故障、高风险设备和薄弱工序。

因此，本项目以文本分类为核心，将设备报修文本转化为结构化结果，为维修调度、停机预警和设备管理提供智能化支撑。

## 3. 项目主要价值

| 业务问题 | 系统解决方式 | 预期价值 |
|---|---|---|
| 人工分单慢 | 自动识别故障大类和处理部门 | 缩短工单响应时间 |
| 工单误派 | 根据文本语义推荐机械、电气、自动化等部门 | 降低返工和转派成本 |
| 严重故障漏判 | 对 P0/P1 风险工单单独预警 | 降低非计划停机损失 |
| 故障统计困难 | 将文本转为结构化标签 | 支撑高频故障分析 |
| 维修经验分散 | 相似工单召回历史处理方案 | 提升维修知识复用 |
| 模型部署成本高 | 蒸馏、量化、剪枝压缩模型 | 提升推理速度，降低部署成本 |

项目可以帮助企业从“被动维修”走向“主动预警”：

```text
单条报修处理 -> 工单自动分派 -> 高风险停机预警 -> 设备故障趋势分析 -> 维修知识沉淀
```

## 4. 项目优势与创新点

### 4.1 项目优势

1. 场景真实  
   设备报修、维修派单、停机预警是制造企业真实存在的业务问题。

2. 工程意义明确  
   输出结果直接服务于维修调度、风险预警和设备管理，不只是做一个模型分数。

3. 标签体系有业务深度  
   不只是简单几分类，而是包含故障大类、风险等级、处理部门等多维度标签。

4. 可展示完整 AI 工程链路  
   包括数据收集、EDA、数据清洗、BERT 微调、模型评估、知识蒸馏、推理服务和人工复核。

5. 简历表达强  
   可以突出“制造业数字化”“工业互联网”“智能运维”“模型压缩与部署”等关键词。

### 4.2 创新点

- 多任务文本分类：同一条报修文本同时预测故障大类、风险等级和处理部门。
- 高风险优先召回：重点优化 P0/P1 停机风险召回率，而不是只看整体准确率。
- 领域化数据构建：结合设备、部件、故障现象、维修部门构建中文工业报修语料。
- BERT + 蒸馏部署：先用 BERT 系列模型保证效果，再用知识蒸馏提升推理速度。
- 人工复核闭环：低置信度样本进入人工复核，修正结果回流到训练集。

## 5. 项目总体实施流程

参考投满分文本分类项目的工程路线，本项目可以按以下阶段推进：

```text
项目背景与业务目标
  -> 数据集收集与构建
  -> EDA 数据探索分析
  -> 数据预处理与清洗
  -> 标签体系设计
  -> Baseline 模型选型
  -> BERT 主模型训练
  -> 模型评估与误差分析
  -> 模型压缩：蒸馏、量化、剪枝
  -> 模型部署：FastAPI / Flask / Streamlit
  -> 人工复核与数据回流
```

其中最重要、最耗时的环节不是模型训练，而是：

> 数据收集、数据预处理、数据清洗、标签检查、样本均衡和数据泄漏控制。

在真实工程中，数据相关工作通常会占项目周期的大部分时间。模型效果能否达到 93% 以上，首先取决于数据质量，而不是单纯换更大的模型。

## 6. 标签体系设计

### 6.1 故障大类

建议设计为 10 类：

| 标签 | 说明 |
|---|---|
| 机械故障 | 主轴、轴承、皮带、导轨、齿轮、链条、卡料、异响 |
| 电气故障 | 电机、变频器、接触器、电源、跳闸、过载、短路 |
| 传感器故障 | 光电、接近开关、温度、压力、液位、编码器、视觉相机 |
| 控制系统故障 | PLC、HMI、伺服、IO模块、通讯、程序、报警无法复位 |
| 工艺参数异常 | 温度、压力、流量、真空度、液位、工艺参数超限 |
| 耗材备件问题 | 刀具、滤芯、皮带、密封圈、喷嘴、润滑油、备件缺料 |
| 安全风险 | 急停、安全门、安全光栅、联锁、漏电、冒烟、夹伤风险 |
| 预防性维护 | 点检、保养、校准、周期更换、计划维护 |
| 质量关联异常 | 尺寸偏差、贴标偏移、灌装不稳、检测误判、废品增加 |
| 公辅系统故障 | 空压站、冷却水、真空站、供气、供电、除尘、排风 |

### 6.2 停机风险等级

| 风险等级 | 含义 |
|---|---|
| P0 | 已停机、无法启动、主线受影响、存在明显安全风险 |
| P1 | 高风险，短时间可能停机，已明显影响产能或质量 |
| P2 | 中风险，暂未停机，但影响效率、节拍或稳定性 |
| P3 | 低风险，计划保养、点检、备件申请、普通维护 |

### 6.3 推荐处理部门

| 部门 | 适用情况 |
|---|---|
| 机械维修 | 机械结构、轴承、皮带、主轴、导轨、卡滞、异响 |
| 电气维修 | 电机、变频器、电源、接触器、跳闸、过载 |
| 自动化工程师 | PLC、HMI、伺服、IO、通讯、程序逻辑 |
| 自动化/仪表 | 传感器、仪表、采集信号、校准问题 |
| 工艺/设备工程师 | 温度、压力、流量、真空度、工艺参数异常 |
| 安全/EHS | 急停、安全门、联锁、漏电、冒烟、夹伤风险 |
| 设备保养/备件管理 | 预防性维护、耗材、滤芯、刀具、备件 |
| 质量/工艺/设备 | 因设备异常导致的质量问题 |
| 公辅/设备维修 | 空压、冷却水、真空、供电、排风等公辅系统 |

## 7. 分类任务形式说明

很多同学第一次看到本项目的数据格式时，容易误解为“一个文本为什么后面跟了好几个类别”。这里需要明确：本项目不是最简单的单一文本分类，而是可以设计成多任务文本分类。

普通新闻分类通常是：

```text
文本 -> 新闻类别
```

本项目的一条设备报修文本，通常需要同时判断多个业务结果：

```text
文本 -> 故障大类
文本 -> 停机风险等级
文本 -> 推荐处理部门
```

也就是说，同一条输入文本会对应三个不同维度的标签。

例如：

```text
二车间1号线包装机光电传感器无信号，检测不到物料，需要尽快处理。
```

模型需要输出：

```text
故障大类：传感器故障
停机风险等级：P1
推荐处理部门：自动化/仪表
```

因此，数据集中一行可以写成：

```text
二车间1号线包装机光电传感器无信号，检测不到物料，需要尽快处理。    传感器故障    P1    自动化/仪表
```

这不是说文本属于三个互相冲突的类别，而是说它有三个业务维度的判断结果。

建议实施顺序：

1. 第一阶段：只做故障大类分类。
2. 第二阶段：增加停机风险等级分类。
3. 第三阶段：增加推荐处理部门分类。
4. 第四阶段：使用一个 BERT 编码器接多个分类头，实现多任务分类。

## 9. 数据收集、EDA、预处理与清洗

这是文本分类项目中最关键的工程环节。很多初学者会把重点放在“换模型”，但真实项目中，模型效果提升往往来自更好的数据处理。

### 9.1 数据收集

数据收集需要明确四个问题：

1. 文本从哪里来：报修描述、点检记录、报警描述、维修记录。
2. 标签从哪里来：维修结论、故障代码、处理部门、停机时长。
3. 数据是否可用：是否有空文本、脏文本、重复文本、标签缺失。
4. 是否可解释：每个标签是否有清晰业务含义。

项目中已构建 22 万+ 条中文语料，后续代码再划分训练集、验证集和测试集。

### 9.2 EDA 数据探索分析

EDA 需要做以下检查：

| 分析项 | 目的 |
|---|---|
| 查看前几条数据 | 确认文本格式和标签格式是否正确 |
| 统计总数据量 | 确认样本量是否满足训练需求 |
| 统计每类样本量 | 检查类别是否均衡 |
| 分析文本长度 | 确定 BERT 最大输入长度 |
| 统计空值和异常值 | 过滤不可训练样本 |
| 检查重复样本 | 避免训练集和测试集泄漏 |
| 绘制类别分布 | 观察长尾类别 |
| 混淆样本分析 | 发现标签边界不清的问题 |

文本长度分析可以统计：

```text
min、max、mean、std、p50、p90、p95、p99
```

然后确定最大文本长度，例如：

```text
max_length = 64 或 128
```

如果文本过长，可以使用 3sigma 原则辅助判断异常长度：

```text
异常阈值 = mean + 3 * std
```

超长文本可以截断、清洗或单独检查。

### 9.3 数据预处理

预处理包括：

- 去除多余空格、换行、制表符。
- 统一中文标点和英文标点。
- 保留关键报警码，例如 E102、TEMP_HIGH、PLC-ERR。
- 统一设备编号表达，例如“1号线”“一号线”“产线1”。
- 统一风险词，例如“停了”“停机”“无法启动”“保护停机”。
- 保留领域关键词，例如 PLC、HMI、伺服、变频器、主轴、光电、急停。

传统模型需要分词：

```text
jieba 分词 -> TF-IDF -> 分类器
```

BERT 模型不强依赖 jieba 分词，通常使用 tokenizer 直接编码：

```text
CLS + text + SEP + PAD
input_ids
attention_mask
labels
```

### 9.4 数据清洗

数据清洗是保证模型准确率的关键。

需要重点处理：

| 问题 | 处理方式 |
|---|---|
| 空文本 | 删除或补充 |
| 标签缺失 | 删除或人工补标 |
| 标签冲突 | 人工复核或规则修正 |
| 重复文本 | 去重 |
| 文本过短 | 检查是否信息不足 |
| 文本过长 | 截断或清洗无关内容 |
| 类别不均衡 | 欠采样、过采样、类别权重 |
| 同模板泄漏 | 按模板或设备分组划分数据 |

### 9.5 标签一致性检查

标签一致性比数据量更重要。例如：

- “PLC报警无法复位”应该稳定归为控制系统故障。
- “光电传感器无信号”应该稳定归为传感器故障。
- “电机过载跳闸”应该稳定归为电气故障。
- “急停复位后无法启动”如果涉及安全回路，应优先归为安全风险。

如果同类文本标签混乱，模型很难达到 93% 以上。

### 9.6 数据划分注意事项

不能简单随机切分所有文本。需要注意：

- 同一模板生成的相似文本不能同时出现在训练集和测试集。
- 同一设备、同一报警码、同一故障组合最好按组划分。
- 测试集应尽量包含人工审核样本。
- 验证集用于调参，测试集只用于最终评估。

## 11. 模型选型

### 11.1 传统机器学习 baseline

可先使用：

- TF-IDF + Logistic Regression
- TF-IDF + Linear SVM
- TF-IDF + RandomForest
- TF-IDF + AdaBoost
- TF-IDF + XGBoost

作用：

- 快速验证数据是否可学。
- 作为 BERT 的对照基线。
- 帮助发现容易混淆的类别。

这类模型优点是训练快、可解释，缺点是对语义理解较弱。

### 11.2 FastText

FastText 适合短文本分类，训练快、推理快。

数据格式类似：

```text
__label__机械故障 CNC机床主轴启动后异响明显，疑似轴承磨损
```

优势：

- 速度很快。
- 可作为轻量化 baseline。
- 对大规模数据训练成本低。

不足：

- 对复杂语义、上下文关系、风险表达理解不如 BERT。

### 11.3 BERT 主模型

推荐主模型：

- `hfl/chinese-roberta-wwm-ext`
- `hfl/chinese-macbert-base`
- `bert-base-chinese`

推荐优先级：

```text
MacBERT / RoBERTa > bert-base-chinese > FastText > TF-IDF
```

BERT 输入格式：

```text
[CLS] 报修文本 [SEP] [PAD] [PAD]
```

核心字段：

```text
input_ids
attention_mask
labels
```

推荐训练策略：

- max_length：64 或 128。
- batch_size：根据显存设置。
- learning_rate：2e-5 到 5e-5。
- epoch：3 到 5。
- early stopping：防止过拟合。
- 使用混淆矩阵进行误差分析。

### 11.4 LLM 辅助

LLM 不建议直接作为主要分类模型，因为部署成本和稳定性较高。但可以用于：

- 辅助生成报修文本表达。
- 辅助难例解释。
- 辅助标签规则检查。
- 辅助构建领域词表。
- 辅助生成测试样例。

注意：LLM 不应该凭空决定标签，标签必须来自业务规则或人工审核。

## 12. 多任务模型设计

可以先做单任务：

```text
报修文本 -> 故障大类
```

再扩展为多任务：

```text
报修文本 -> 故障大类 + 风险等级 + 推荐处理部门
```

模型结构：

```text
报修文本
  ↓
BERT Encoder
  ↓
[CLS] 向量
  ↓
故障大类分类头
风险等级分类头
处理部门分类头
```

损失函数：

```text
总损失 = 故障大类损失 + 风险等级损失 + 处理部门损失
```

如果 P0/P1 样本较少，可以对风险等级任务加入类别权重，提高高风险样本召回率。

## 13. 如何让准确率达到 93% 以上

准确率达到 93% 以上，不能只靠模型，核心依赖数据质量。

### 13.1 数据层面

1. 标签边界清晰  
   例如“PLC报警”归控制系统，“光电无信号”归传感器，“变频器报警”归电气。

2. 类别均衡  
   每类样本量尽量接近，避免模型偏向大类。

3. 严格去重  
   相似模板不能同时进入训练集和测试集。

4. 难例增强  
   加入容易混淆的文本，例如“伺服报警”可能属于电气，也可能属于控制系统。

5. 测试集人工抽检  
   测试集必须保证标签可靠，否则 93% 指标没有意义。

### 13.2 模型层面

- 使用中文 RoBERTa 或 MacBERT 微调。
- 控制 max_length，避免无效 padding 过多。
- 使用学习率 2e-5 到 5e-5。
- 使用 early stopping。
- 对 P0/P1 风险类别加权。
- 根据混淆矩阵补充样本。

### 13.3 工程策略

- 低置信度进入人工复核。
- Top-2 结果辅助人工判断。
- 高风险类别宁可多报，不能漏报。
- 人工修正样本回流训练集。

推荐目标：

| 指标 | 目标 |
|---|---:|
| 故障大类 Accuracy | 93%+ |
| Macro-F1 | 90%+ |
| P0/P1 Recall | 95%+ |
| Top-2 Accuracy | 96%+ |
| 蒸馏后准确率下降 | 控制在 1% 到 3% |

## 14. 模型压缩

模型压缩的目标是：

> 在尽量保持准确率的前提下，提高推理速度，降低部署成本。

### 14.1 量化

量化是把模型参数从 FP32 转为 INT8 或 INT4。

常见方式：

- 训练后量化 PTQ
- 量化感知训练 QAT
- 动态量化

适合本项目的做法：

```text
BERT Teacher -> 蒸馏 Student -> 动态量化 -> 推理部署
```

### 14.2 知识蒸馏

Teacher 模型：

```text
Chinese RoBERTa / MacBERT
```

Student 模型：

```text
TinyBERT / DistilBERT / MiniLM / 4层中文BERT
```

蒸馏方式：

- 硬标签蒸馏：真实标签指导学生模型。
- 软标签蒸馏：教师模型输出概率分布指导学生模型。
- 混合蒸馏：CE + KL。

蒸馏损失：

```text
总损失 = α * CE真实标签损失 + β * KL软标签蒸馏损失
```

### 14.3 模型剪枝

剪枝可以减少模型中的冗余连接或结构。

可选方向：

- 非结构化剪枝
- 结构化剪枝
- 全局剪枝
- 特定层剪枝

实际项目中，优先级建议：

```text
知识蒸馏 > 动态量化 > ONNX Runtime > 剪枝
```

剪枝实现复杂度较高，可以作为扩展亮点，不一定作为第一版必须完成的功能。

### 14.4 压缩后评估

| 指标 | Teacher BERT | Student 蒸馏模型 |
|---|---:|---:|
| Accuracy | 目标 93%+ | 尽量保持 91% 到 93% |
| Macro-F1 | 高 | 小幅下降可接受 |
| P0/P1 Recall | 重点保障 | 不能明显下降 |
| 模型体积 | 较大 | 明显减小 |
| 推理速度 | 较慢 | 明显提升 |

## 15. 模型部署与工程落地

### 15.1 服务端接口

可使用 FastAPI 或 Flask。

推荐 FastAPI，因为接口文档和异步能力更友好。

接口设计：

```text
POST /predict
输入：报修文本
输出：故障大类、风险等级、处理部门、置信度、是否复核
```

服务端流程：

1. 导入依赖包。
2. 加载 tokenizer 和模型。
3. 定义文本预处理函数。
4. 定义预测函数。
5. 定义接口响应格式。
6. 启动服务。

### 15.2 客户端演示

可选：

- Streamlit
- Gradio
- 简单 HTML 页面

推荐 Streamlit 做项目演示，因为实现快、适合展示单条预测和批量预测。

### 15.3 工程模块

| 模块 | 功能 |
|---|---|
| 数据清洗模块 | 清洗特殊符号、设备编号、报警码 |
| 模型推理模块 | 调用分类模型输出结果 |
| 规则增强模块 | 对急停、冒烟、漏电等强风险词直接提升风险等级 |

### 15.4 推理速度优化

为了保证运行速度不能太慢，可以采用：

- 蒸馏小模型。
- ONNX Runtime。
- 动态量化。
- 批量推理。
- 缓存重复文本预测结果。
- 限制最大文本长度为 64 或 128。
- 低并发场景 CPU 部署，高并发场景 GPU 或推理服务化部署。

## 16. 项目实施步骤

    针对混淆类别补充样本。

11. 模型压缩  
    知识蒸馏、动态量化、ONNX Runtime。

12. 模型部署  
    FastAPI 提供接口，Streamlit 或 Gradio 做演示。

    低置信度样本回流训练集。

## 18. 面试注意事项

### 18.1 数据真实性

不要说“这是某工厂真实内部工单”，除非确实有企业授权。推荐说法：

> 真实制造业 CMMS/EAM 工单通常不公开，因此我采用公开工业故障知识和制造业设备运维规则构建中文报修语料。标签来自设备、部件、故障现象、风险等级和处理部门的业务规则，不是大模型凭空生成。后续如果进入企业落地，可以替换为真实工单数据并继续微调。

### 18.2 关于 93% 准确率

不能只说“准确率 93%”，需要说明：

- 是故障大类分类准确率，还是多任务综合准确率。
- 测试集是否去重。
- 类别是否均衡。
- 是否存在模板泄漏。
- P0/P1 召回率是多少。

推荐表达：

> 在去重后的 10 类故障大类测试集上，BERT 模型准确率目标为 93%+；同时重点关注 P0/P1 高风险样本召回率，避免严重停机风险漏判。

### 18.3 关于模型压缩

面试官可能问：为什么不用原始 BERT？

回答要点：

> 原始 BERT 效果好，但企业内网或本地服务器部署时要考虑模型体积、推理速度和并发成本。因此使用知识蒸馏和量化训练轻量 Student 模型，在尽量保持分类效果的同时提升推理速度，更符合工业场景部署要求。

### 18.4 关于项目边界

该项目是文本分类与风险预警，不等同于完整预测性维护系统。它不直接预测设备剩余寿命，而是对报修文本进行故障归类和风险识别。如果后续接入传感器时间序列、报警日志和停机记录，可以进一步扩展到预测性维护。

## 20. 总结

本项目的核心不是“做一个文本分类模型”，而是围绕制造业设备运维中的真实业务问题，构建设备报修文本智能分类、停机风险预警和维修部门推荐系统。

项目价值可以概括为：

```text
减少人工分单时间 + 降低误派率 + 提前识别停机风险 + 提升维修响应效率 + 支撑设备运维数字化
```

最终方案需要体现三条主线：

1. 数据主线：数据收集、EDA、清洗、预处理、标签检查、类别均衡。
2. 模型主线：TF-IDF/FastText baseline、BERT 主模型、多任务分类。
3. 工程主线：蒸馏压缩、量化提速、FastAPI 部署、人工复核闭环。

如果这三条主线都做完整，项目就不再是简单的“文本分类练习”，而是一个具有工程意义、业务价值和简历展示价值的制造业智能运维 NLP 项目。
